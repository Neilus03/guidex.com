GuideX

Guided Synthetic Data Generation for Zero-Shot Information Extraction

Neil De La Fuente · Oscar Sainz · Iker García-Ferrero · Eneko Agirre
HiTZ - EHU

ACL Findings 2025

Abstract

Information Extraction (IE) systems are traditionally domain-specific, requiring costly adaptation that involves expert schema design, data annotation, and model training. While Large Language Models have shown promise in zero-shot IE, performance degrades significantly in unseen domains where label definitions differ. This paper introduces GuideX, a novel method that automatically defines domain-specific schemas, infers guidelines, and generates synthetically labeled instances, allowing for better out-of-domain generalization. Fine-tuning LLaMa 3.1 with GuideX sets a new state-of-the-art across seven zero-shot Named Entity Recognition benchmarks. Models trained with GuideX gain up to 7 F1 points over previous methods without human-labeled data, and nearly 3 F1 points higher when combined with it. Models trained on GuideX demonstrate enhanced comprehension of complex, domain-specific annotation schemas.

Methodology  4 stages + QA

GuideX is a fully-automatic pipeline that converts raw documents into high-fidelity (guidelines + annotations) pairs. Each stage runs as a single prompt to the same 70B LLaMA-3.1 model, chaining its own output into the next step.

  1. Document Summarisation  Step ①
    The LLM receives a full article (≈1-3 k words) and is forced—via a token-budget and bullet template—to extract only atomic facts.
    • ✓ collapses core arguments, dates, actors, quantities
    • ✓ discards narrative fluff & irrelevancies
  2. Structured Sketch  Step ②
    Those bullets are rewritten into a {...} JSON stub:
    • Keys  = coarse entity/slot names inferred on-the-fly
    • Values = the shortest source-aligned span that realises the fact
    • Merging logic groups near-duplicates to keep the schema minimal
  3. Guideline Drafting  Step ③
    From the JSON, the model autowrites a Python @dataclass file:
    • Each key → a separate dataclass with a rich docstring (definition + edge-cases)
    • Fields typed as str, int, List[str], Optional[⋅] etc. so the file is executable
    • Naming heuristics strip plurals, camel-case and jargon to yield reusable labels (“MachineLearningFramework” → “Framework”)
  4. Instance Extraction  Step ④

Automated Quality-Assurance

The generated file is import-ed inside a tiny unit-test. Anything that fails to compile or violates type hints is binned. This removes noisy samples and guarantees that every surviving pair is schema-consistent and executable.
